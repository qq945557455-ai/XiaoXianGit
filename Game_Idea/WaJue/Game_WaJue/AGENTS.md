# AGENTS.md — 大语言模型维基维护规范

本知识库采用 Andrej Karpathy 的 **大语言模型维基（LLM Wiki）** 模式：把知识库当作可持续维护的 Markdown 代码库。用户负责提供资料、提出问题和判断方向；大语言模型负责阅读、摘要、归档、交叉链接、更新索引、记录日志和定期体检。

## 三层架构

1. **原始资料/**
   - 原始资料层，视为事实来源。
   - 可以新增资料，但除非用户明确要求，后续不要改写已有原始资料。
   - 网页资料优先保存为“来源登记卡”，包含网址、访问日期、主题、摘要和后续处理状态；避免无许可全文搬运。
   - 图片和附件放入 `原始资料/附件/`，并使用 Obsidian 的 `![[...]]` 嵌入。

2. **知识库/**
   - 大语言模型维护的知识层。
   - 包含来源摘要、概念页、实体页、项目页、问题页、综合分析页、索引和日志。
   - 可以由大语言模型创建和更新，但每次改动都要尽量保持小而清晰。

3. **AGENTS.md**
   - 结构和流程规范层。
   - 当实践中发现更好的分类、模板或流程时，先向用户说明，再更新本文件。

## 目录约定

- `知识库/索引.md`：内容索引。回答问题或更新知识库前，优先阅读。
- `知识库/日志.md`：追加式操作日志。每条记录使用 `## [YYYY-MM-DD] 类型 | 标题` 格式。
- `知识库/来源摘要/`：对单个来源的摘要和加工记录。
- `知识库/概念/`：可复用概念、方法论、设计原则。
- `知识库/实体/`：人物、组织、工具、地点、IP、项目等实体。
- `知识库/项目/`：本游戏项目、系统、玩法、内容管线等项目页。
- `知识库/问题/`：由提问沉淀出的分析页、对比页、决策页。
- `知识库/模板/`：新页面模板。

## 页面格式

优先使用 YAML 属性区：

```yaml
---
type: concept | entity | source-summary | project | question | index | log
status: seed | active | stable | needs-review
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: []
sources: []
---
```

正文建议包含：

- `## 摘要`：3–6 行核心结论。
- `## 关键点`：结构化要点。
- `## 关联`：使用 `[[知识库/...]]` 链接到相关页面。
- `## 待办 / 问题`：仍需验证或继续研究的事项。
- `## 来源`：外部链接或 `[[原始资料/...]]` 来源卡。

## 工作流

### 摄入：处理新资料

当用户提供文章、视频、论文、图片、聊天记录或设计文档时：

1. 将其登记到 `原始资料/来源登记/` 或引用已有原始文件。
2. 阅读来源；必要时单独查看嵌入图片。
3. 在 `知识库/来源摘要/` 写来源摘要页。
4. 更新相关 `知识库/概念/`、`知识库/实体/`、`知识库/项目/` 页面。
5. 如发现旧说法被新资料挑战，明确标记“冲突/更新”。
6. 更新 `知识库/索引.md`。
7. 追加 `知识库/日志.md`。

### 查询：回答和沉淀问题

1. 先读 `知识库/索引.md`，再读相关页面。
2. 回答时给出基于知识库页面的引用。
3. 如果问题产生了有复用价值的分析、对比、决策或清单，沉淀到 `知识库/问题/` 或相关主题页。
4. 更新索引和日志。

### 体检：知识库健康检查

定期检查：

- 孤立页面和缺少反向链接的页面。
- 已提到但没有独立页面的重要概念。
- 重复页面、命名漂移、分类漂移。
- 新旧资料之间的冲突。
- 索引遗漏、日志遗漏。
- 可以通过继续搜索或补充资料解决的数据缺口。

## 命名和链接

- 文件名优先使用清晰中文；必要的技术名词、工具名和专有名词可以保留英文。
- 跨页面引用使用 Obsidian 维基链接：`[[知识库/概念/大语言模型维基]]`。
- 不确定但值得创建的概念可先用维基链接标记，后续补页。
- 避免一次性创建大量空页面；优先创建能承载真实信息的页面。

## 当前项目特化

此知识库位于游戏项目 `Game_WaJue`。除通用知识管理外，建议把后续游戏资料逐步沉淀为：

- 世界观、角色、地点、阵营：`知识库/实体/`
- 核心循环、系统、关卡、资源、经济：`知识库/项目/`
- 竞品、研究文章、设计资料：`原始资料/来源登记/` + `知识库/来源摘要/`
- 设计问题和决策记录：`知识库/问题/`

## 职能资产归档规则

后续内容先判断主责职能，策划、程序、美术、质量管理 / QA 各自的资料、参考图、素材、说明页和交付物，都放到对应职能文件夹下；跨职能内容放到公共文件夹：

- 知识库页面：放入 `知识库/项目/职能分工/策划/`、`知识库/项目/职能分工/程序/`、`知识库/项目/职能分工/美术/`、`知识库/项目/职能分工/质量管理/` 或 `知识库/项目/职能分工/公共/` 的相应子目录。
- 原始附件：放入 `原始资料/附件/职能分工/策划/`、`原始资料/附件/职能分工/程序/`、`原始资料/附件/职能分工/美术/`、`原始资料/附件/职能分工/质量管理/` 或 `原始资料/附件/职能分工/公共/` 的相应子目录。
- QA 环节资料：验收清单、测试用例、缺陷记录、回归验证、质量风险放入 `知识库/项目/职能分工/质量管理/`。
- 跨职能资料：公共排期、评审、里程碑、会议纪要、跨职能任务和跨职能决策放入 `知识库/项目/职能分工/公共/`，其他职能页面用 Obsidian 链接引用，不重复复制。

## 需求看板使用规则

后续不再为人员管理维护流程类白板。管理者需求统一记录到 `知识库/项目/职能分工/公共/任务清单/需求看板.md`，用于快速查看和跳转。

需求看板只保留：

- 各职能被下达的需求内容。
- 产出结果大纲。
- 完成进度。
- 评分：低 / 中 / 高。
- 对应需求相关的跳转链接。

流程、周节奏、风险清单和详细规则应写入对应 Markdown 笔记，不放入需求看板主体。
